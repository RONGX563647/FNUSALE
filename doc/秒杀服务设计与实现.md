## 1. 设计原理

### 1.1 秒杀业务特点

| 特点 | 说明 | 技术挑战 |
|------|------|----------|
| **瞬时高并发** | 活动开始瞬间流量激增 | 系统承载能力、数据库压力 |
| **库存有限** | 通常 10-50 件 | 防止超卖、数据一致性 |
| **时间限制** | 限时抢购 | 精准的时间控制 |
| **公平性** | 每人限购一件 | 用户去重 |
| **高可用** | 不能宕机 | 服务稳定性 |

### 1.2 设计目标

```
┌─────────────────────────────────────────────────────────────┐
│                      秒杀系统设计目标                         │
├─────────────────────────────────────────────────────────────┤
│  1. 高性能：支持 QPS 500+（校园规模）                         │
│  2. 防超卖：库存扣减绝对准确                                  │
│  3. 高可用：服务降级、限流保护                                │
│  4. 最终一致性：Redis 与数据库数据最终一致                    │
│  5. 用户体验：快速响应、结果可查询                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 核心设计思路

#### 三层防护架构

```
                    ┌──────────────┐
                    │   客户端     │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  CDN/静态化  │  ← 第一层：静态资源加速
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │    Gateway   │  ← 第二层：限流、鉴权
                    └──────┬───────┘
                           │
         ┌─────────────────┼─────────────────┐
         │                 │                 │
   ┌─────▼─────┐    ┌─────▼─────┐    ┌─────▼─────┐
   │   Redis   │    │  Sentinel │    │   MQ      │  ← 第三层：缓存+限流+异步
   │  库存扣减  │    │   限流    │    │ 订单创建  │
   └───────────┘    └───────────┘    └───────────┘
```

#### 为什么这样设计？

| 层级 | 技术 | 原因 |
|------|------|------|
| **Redis 缓存** | 库存预热、原子扣减 | Redis 内存操作，QPS 可达 10W+ |
| **Lua 脚本** | 原子性保证 | 避免多命令间的竞态条件 |
| **RocketMQ** | 异步解耦 | 削峰填谷，保护订单服务 |
| **本地消息表** | 最终一致性 | 保证 Redis 扣减与 MQ 发送的一致性 |
| **Sentinel** | 限流保护 | 防止系统过载 |

### 3.1 Lua 脚本原子扣减

**文件**: [SeckillLuaScript.java](../backend/fnusale-marketing/src/main/java/com/fnusale/marketing/script/SeckillLuaScript.java)

本项目实现了 4 个 Lua 脚本，每个脚本都有特定的用途和返回值设计。

#### 3.1.1 预热并扣减脚本（核心）

这是项目中最核心的脚本，解决了库存预热和扣减之间的竞态条件问题。

```java
/**
 * 预热并扣减库存脚本（原子操作）
 * 解决预热后扣减间的竞态条件
 * KEYS[1]: 库存key
 * ARGV[1]: 预热库存值（当key不存在时使用）
 * ARGV[2]: 过期时间（秒）
 * 返回值说明：
 *  正数: 扣减成功，返回剩余库存
 *  0: 库存不足（扣减后为0）
 *  -1: 参数错误
 */
public static final String WARM_UP_AND_DECREMENT_SCRIPT = """
        local stock = redis.call('GET', KEYS[1])
        if stock == false then
            -- 库存未预热，使用传入的值初始化
            if ARGV[1] and tonumber(ARGV[1]) > 0 then
                redis.call('SET', KEYS[1], ARGV[1])
                if ARGV[2] and tonumber(ARGV[2]) > 0 then
                    redis.call('EXPIRE', KEYS[1], ARGV[2])
                end
                stock = ARGV[1]
            else
                return -1
            end
        end

        local remain = tonumber(stock)
        if remain <= 0 then
            return 0
        end

        return redis.call('DECR', KEYS[1])
        """;
```

**代码逐行解析**：

```lua
-- 第1行：获取当前库存
local stock = redis.call('GET', KEYS[1])
-- 如果库存key不存在，说明未预热
if stock == false then
    -- 第4-7行：检查预热参数是否有效
    if ARGV[1] and tonumber(ARGV[1]) > 0 then
        -- 第6行：设置初始库存
        redis.call('SET', KEYS[1], ARGV[1])
        -- 第7-9行：设置过期时间（防止数据永久占用内存）
        if ARGV[2] and tonumber(ARGV[2]) > 0 then
            redis.call('EXPIRE', KEYS[1], ARGV[2])
        end
        -- 第11行：更新stock变量为初始值
        stock = ARGV[1]
    else
        -- 第13行：参数无效，返回错误
        return -1
    end
end

-- 第16行：将字符串转为数字
local remain = tonumber(stock)
-- 第17-19行：检查库存是否充足
if remain <= 0 then
    return 0
end

-- 第22行：扣减库存并返回剩余数量
return redis.call('DECR', KEYS[1])
```

**返回值处理逻辑**（[SeckillServiceImpl.java#L224-L231](../backend/fnusale-marketing/src/main/java/com/fnusale/marketing/service/impl/SeckillServiceImpl.java#L224-L231)）：

```java
    log.error("秒杀库存扣减参数错误: activityId={}", activityId);
    throw new BusinessException(500, "系统繁忙，请稍后重试");
```

**设计要点**:

| 要点 | 说明 |
|------|------|
| **原子性** | 整个脚本在 Redis 中原子执行，不会被其他命令打断 |
| **预热+扣减合并** | 解决了两步操作间的竞态条件 |
| **返回值明确** | 正数=剩余库存，0=库存不足，-1=参数错误 |
| **自动过期** | 防止 Redis 数据永久占用内存 |

**为什么用 Lua 脚本？**

```
❌ 不使用 Lua 的问题：

时间线: ─────────────────────────────────►

线程A: GET stock → 返回 null（未预热）
线程B: GET stock → 返回 null（未预热）
线程A: SET stock 50 → 预热成功
线程B: SET stock 50 → 覆盖了A的预热！
线程A: DECR stock → 返回 49
线程B: DECR stock → 返回 48  ← 数据不一致！

✅ 使用 Lua 的解决方案：

线程A: 执行 Lua 脚本 → 原子操作 → 预热+扣减 → 返回 49
线程B: 执行 Lua 脚本 → 原子操作 → 直接扣减 → 返回 48
```

### 3.2 秒杀核心方法

**文件**: [SeckillServiceImpl.java](../backend/fnusale-marketing/src/main/java/com/fnusale/marketing/service/impl/SeckillServiceImpl.java)

#### 3.2.1 完整代码实现

```java
@Override
@Transactional(rollbackFor = Exception.class)
public Long joinSeckill(Long userId, Long activityId) {
    // ========== 第1步：查询活动信息 ==========
    SeckillActivity activity = activityMapper.selectById(activityId);
    if (activity == null) {
        throw new BusinessException(5001, "秒杀活动不存在");
    }

    // ========== 第2步：检查活动状态 ==========
    if (MarketingConstants.SECKILL_STATUS_NOT_START.equals(activity.getActivityStatus())) {
        throw new BusinessException(5002, "秒杀活动尚未开始");
    }
    if (MarketingConstants.SECKILL_STATUS_END.equals(activity.getActivityStatus())) {
        throw new BusinessException(5003, "秒杀活动已结束");
    }

    // ========== 第3步：检查用户是否已参与（Redis Set） ==========
    String boughtKey = MarketingConstants.SECKILL_USER_BOUGHT_PREFIX + activityId;
    Boolean hasBought = redisTemplate.opsForSet().isMember(boughtKey, userId.toString());
    if (Boolean.TRUE.equals(hasBought)) {
        throw new BusinessException(5005, "您已参与过该秒杀");
    }

    // ========== 第4步：Lua 脚本原子预热并扣减库存 ==========
    String stockKey = MarketingConstants.SECKILL_STOCK_KEY_PREFIX + activityId;
    
    // 计算过期时间（活动结束时间 - 当前时间）
    long expireSeconds = Duration.between(LocalDateTime.now(), activity.getEndTime()).getSeconds();
    if (expireSeconds <= 0) {
        // 如果活动已结束，使用默认过期时间30分钟
        expireSeconds = MarketingConstants.SECKILL_STOCK_PRELOAD_MINUTES * 60L;
    }

    // 使用 Lua 脚本原子预热并扣减库存
    Long result = redisTemplate.execute(
            warmUpAndDecrementScript,
            Collections.singletonList(stockKey),  // KEYS[1]: 库存key
            activity.getRemainStock().toString(),  // ARGV[1]: 预热库存值
            String.valueOf(expireSeconds)  // ARGV[2]: 过期时间（秒）
    );

    // 处理 Lua 脚本返回值
    // result: -1=参数错误, 0=库存不足, 正数=扣减成功返回剩余库存
    if (result == null || result == -1) {
        log.error("秒杀库存扣减参数错误: activityId={}", activityId);
        throw new BusinessException(500, "系统繁忙，请稍后重试");
    }

    if (result == 0) {
        throw new BusinessException(5004, "秒杀库存不足");
    }

    // ========== 第5步：标记用户已购买 ==========
    redisTemplate.opsForSet().add(boughtKey, userId.toString());
    // 设置过期时间为活动结束时间
    if (expireSeconds > 0) {
        redisTemplate.expire(boughtKey, expireSeconds, TimeUnit.SECONDS);
    }

    // ========== 第6步：生成秒杀订单事件 ==========
    String eventId = UUID.randomUUID().toString();
    
    SeckillOrderEvent orderEvent = SeckillOrderEvent.builder()
            .userId(userId)
            .activityId(activityId)
            .productId(activity.getProductId())
            .seckillPrice(activity.getSeckillPrice())
            .quantity(1)
            .eventId(eventId)
            .seckillTime(LocalDateTime.now())
            .build();

    // ========== 第7步：本地消息表 + MQ 发送（确保最终一致性） ==========
    try {
        // 7.1 将消息内容序列化为 JSON
        String messageContent = objectMapper.writeValueAsString(orderEvent);

        // 7.2 保存到本地消息表（在同一事务内）
        LocalMessage localMessage = LocalMessage.builder()
                .messageId(eventId)
                .messageType(LocalMessageStatus.MessageType.SECKILL_ORDER)
                .topic(RocketMQConstants.SECKILL_ORDER_TOPIC)
                .tag(RocketMQConstants.SECKILL_ORDER_TAG_CREATE)
                .messageContent(messageContent)
                .status(LocalMessageStatus.PENDING)  // 初始状态：待发送
                .retryCount(0)
                .maxRetryCount(LocalMessageStatus.DEFAULT_MAX_RETRY_COUNT)
                .nextRetryTime(LocalDateTime.now())
                .createTime(LocalDateTime.now())
                .updateTime(LocalDateTime.now())
                .build();
        localMessageMapper.insert(localMessage);

        // 7.3 尝试发送 MQ 消息（同步发送）
        boolean sent = eventPublisher.sendSync(
                RocketMQConstants.SECKILL_ORDER_TOPIC,
                RocketMQConstants.SECKILL_ORDER_TAG_CREATE,
                orderEvent
        );

        if (sent) {
            // 发送成功，更新消息状态为 SENT
            localMessageMapper.updateToSent(localMessage.getId());
            log.info("秒杀订单消息发送成功: eventId={}", eventId);
        } else {
            // 发送失败，消息保持 PENDING 状态，由定时任务重试
            log.warn("秒杀订单消息发送失败，将由定时任务重试: eventId={}", eventId);
        }
    } catch (Exception e) {
        log.error("秒杀订单处理异常: userId={}, activityId={}", userId, activityId, e);
        // 注意：此处不抛出异常，因为 Redis 库存已扣减，用户已获得秒杀资格
        // 消息将由定时任务重试发送
    }

    log.info("用户 {} 参与秒杀活动 {} 成功，订单创建中", userId, activityId);
    return null; // 订单正在创建中，用户可轮询查询结果
}
```

```java
@Override
public void processPendingMessages() {
    LocalDateTime now = LocalDateTime.now();
    List<LocalMessage> messages = localMessageMapper.selectPendingMessages(now, 100);

    for (LocalMessage message : messages) {
        try {
            // 发送消息到MQ
            eventPublisher.sendRawMessage(message.getTopic(), message.getTag(), message.getMessageContent());

            // 更新为已发送
            localMessageMapper.updateToSent(message.getId());
            log.info("消息发送成功: messageId={}", message.getMessageId());
        } catch (Exception e) {
            log.error("消息发送失败: messageId={}, error={}", message.getMessageId(), e.getMessage());

            // 计算下次重试时间（指数退避）
            int retryIndex = Math.min(message.getRetryCount(), LocalMessageStatus.RETRY_INTERVALS.length - 1);
            LocalDateTime nextRetryTime = LocalDateTime.now().plusMinutes(LocalMessageStatus.RETRY_INTERVALS[retryIndex]);

            // 更新为失败状态
            localMessageMapper.updateToFailed(message.getId(), nextRetryTime);

            // 如果超过最大重试次数，记录错误日志
            if (message.getRetryCount() + 1 >= message.getMaxRetryCount()) {
                log.error("消息超过最大重试次数，需要人工处理: messageId={}, content={}",
                        message.getMessageId(), message.getMessageContent());
            }
        }
    }
}
```

**教学要点**：
- 每次最多处理 100 条消息，避免单次处理过多
- 使用指数退避策略：1分钟 → 2分钟 → 5分钟 → 10分钟 → 30分钟
- 超过最大重试次数后需要人工介入
- 每次重试都会更新 `next_retry_time` 字段

#### 3.3.6 重试机制详解

**指数退避策略**：

```
重试次数    重试间隔    累计时间
─────────────────────────────
   0         1分钟       1分钟
   1         2分钟       3分钟
   2         5分钟       8分钟
   3        10分钟      18分钟
   4        30分钟      48分钟
   5       人工介入      ─────
```

**为什么使用指数退避？**

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **固定间隔** | 简单 | 可能频繁重试导致系统压力 | 临时性故障 |
| **线性增长** | 逐步增加重试间隔 | 增长较慢 | 一般性故障 |
| **指数退避** ✅ | 快速减少重试频率 | 恢复较慢 | 持续性故障 |

**代码实现**（[SeckillServiceImpl.java#L91-L111](../backend/fnusale-marketing/src/main/java/com/fnusale/marketing/service/impl/SeckillServiceImpl.java#L91-L111)）：

```java
/**
 * 商品信息本地缓存
 * 缓存秒杀商品信息，减少远程调用
 */
private Cache<Long, ProductVO> productCache;

@PostConstruct
public void init() {
    // 初始化 Redis 脚本
    decrementStockScript = new DefaultRedisScript<>();
    decrementStockScript.setScriptText(SeckillLuaScript.DECREMENT_STOCK_SCRIPT);
    decrementStockScript.setResultType(Long.class);

    warmUpStockScript = new DefaultRedisScript<>();
    warmUpStockScript.setScriptText(SeckillLuaScript.WARM_UP_STOCK_SCRIPT);
    warmUpStockScript.setResultType(Long.class);

    warmUpAndDecrementScript = new DefaultRedisScript<>();
    warmUpAndDecrementScript.setScriptText(SeckillLuaScript.WARM_UP_AND_DECREMENT_SCRIPT);
    warmUpAndDecrementScript.setResultType(Long.class);

    // 初始化商品信息缓存
    productCache = Caffeine.newBuilder()
            .maximumSize(1000)              // 最大缓存 1000 条
            .expireAfterWrite(Duration.ofMinutes(10))  // 写入后 10 分钟过期
            .recordStats()                  // 开启统计
            .build();
}
```

**Caffeine 配置参数详解**：

| 参数 | 值 | 说明 |
|------|-----|------|
| **maximumSize** | 1000 | 最大缓存条目数，超过后使用 LRU 策略淘汰 |
| **expireAfterWrite** | 10分钟 | 写入后10分钟过期，保证数据新鲜度 |
| **recordStats** | true | 开启统计，可监控缓存命中率 |

**为什么选择 10 分钟过期？**

```
考虑因素：
1. 商品信息变更频率较低（价格、名称等）
2. 秒杀活动通常持续 2 小时以内
3. 10 分钟过期平衡了性能和数据一致性

如果过期时间太短（如1分钟）：
  → 缓存命中率低，频繁调用商品服务
  → 增加网络开销和响应时间

如果过期时间太长（如1小时）：
  → 商品信息变更后缓存未及时更新
  → 可能显示过时的商品信息
```

```java
        var result = productClient.getProductById(key);
        if (result != null && result.isSuccess() && result.getData() != null) {
            // 第5行：返回商品信息，Caffeine 会自动缓存
            return result.getData();
        }
    } catch (Exception e) {
        // 第8行：异常处理，记录警告日志
        log.warn("获取秒杀商品信息失败: productId={}", key);
    }
    // 第10行：返回 null，表示加载失败
    return null;
});
```

**教学要点**：
- `productCache.get(key, loader)` 是 Caffeine 的核心 API
- 如果缓存命中，直接返回缓存值
- 如果缓存未命中，执行 loader 函数加载数据并缓存
- loader 函数返回 null 时，不会缓存（避免缓存空值）

### 3.5 批量查询优化

**目的**: 解决 N+1 查询问题，提升查询性能

**相关文件**:

#### 3.5.1 N+1 问题分析

**优化前（N+1 问题）**：

```java
// ❌ 错误示范：每次循环都查询一次数据库
List<SeckillActivityVO> voList = getSeckillList(userId);

for (SeckillActivityVO vo : voList) {
    // 每次循环都执行一次数据库查询
    int count = reminderMapper.countByUserAndActivity(userId, vo.getId());
    vo.setReminded(count > 0);
}

// 假设有 10 个活动，就会执行 10 次数据库查询
// 1 次查询活动列表 + 10 次查询提醒状态 = 11 次数据库查询
```

**问题分析**：

```
N+1 问题：
- N = 活动数量（如 10 个）
- 1 = 查询活动列表（1 次）
- 总查询次数 = 1 + N = 11 次

性能影响：
- 数据库连接池压力大
- 网络往返次数多
- 响应时间长（11 次查询 × 10ms = 110ms）
```

#### 3.5.2 优化方案：批量查询

**优化后（批量查询）**：

```java
// ✅ 正确示范：批量查询提醒状态
List<SeckillActivityVO> voList = getSeckillList(userId);

// 提取所有活动 ID
List<Long> activityIds = voList.stream()
        .map(SeckillActivityVO::getId)
        .collect(Collectors.toList());

// 一次查询获取所有活动的提醒状态
Set<Long> remindedActivityIds = reminderMapper.selectRemindedActivityIds(userId, activityIds);

// 设置提醒状态
voList.forEach(vo -> vo.setReminded(remindedActivityIds.contains(vo.getId())));

// 只执行 2 次数据库查询
// 1 次查询活动列表 + 1 次查询提醒状态 = 2 次数据库查询
```

**性能提升**：

```
优化前：11 次查询 × 10ms = 110ms
优化后： 2 次查询 × 10ms = 20ms
性能提升：5.5 倍
```

```java
@Override
public List<SeckillActivityVO> getSeckillList(Long userId) {
    String cacheKey = MarketingConstants.SECKILL_TODAY_KEY;
    List<SeckillActivityVO> voList;

    // 尝试从缓存获取
    try {
        String cachedList = redisTemplate.opsForValue().get(cacheKey);
        if (cachedList != null) {
            voList = objectMapper.readValue(cachedList,
                    objectMapper.getTypeFactory().constructCollectionType(List.class, SeckillActivityVO.class));
        } else {
            // 查询数据库
            List<SeckillActivity> activities = activityMapper.selectActiveActivities();
            voList = activities.stream()
                    .map(this::convertToVO)
                    .collect(java.util.stream.Collectors.toList());

            // 缓存5分钟
            if (!voList.isEmpty()) {
                redisTemplate.opsForValue().set(cacheKey,
                        objectMapper.writeValueAsString(voList),
                        Duration.ofMinutes(5));
            }
        }
    } catch (Exception e) {
        log.warn("读取活动列表缓存失败，从数据库查询", e);
        List<SeckillActivity> activities = activityMapper.selectActiveActivities();
        voList = activities.stream()
                .map(this::convertToVO)
                .collect(java.util.stream.Collectors.toList());
    }

    // ========== 批量查询提醒状态（解决 N+1 问题） ==========
    if (userId != null && !voList.isEmpty()) {
        // 提取所有活动 ID
        List<Long> activityIds = voList.stream()
                .map(SeckillActivityVO::getId)
                .collect(java.util.stream.Collectors.toList());
        
        // 批量查询提醒状态
        java.util.Set<Long> remindedActivityIds = reminderMapper.selectRemindedActivityIds(userId, activityIds);
        
        // 设置提醒状态
        voList.forEach(vo -> vo.setReminded(remindedActivityIds.contains(vo.getId())));
    } else {
        voList.forEach(vo -> vo.setReminded(false));
    }
```

```java
@Override
public List<TodaySeckillVO> getTodaySeckills(Long userId) {
    List<SeckillActivity> activities = activityMapper.selectTodayActivities();

    if (activities.isEmpty()) {
        return new ArrayList<>();
    }

    // 转换为 VO 列表
    List<SeckillActivityVO> voList = activities.stream()
            .map(this::convertToVO)
            .collect(java.util.stream.Collectors.toList());

    // ========== 批量查询提醒状态（解决 N+1 问题） ==========
    if (userId != null && !voList.isEmpty()) {
        List<Long> activityIds = voList.stream()
                .map(SeckillActivityVO::getId)
                .collect(java.util.stream.Collectors.toList());
        java.util.Set<Long> remindedActivityIds = reminderMapper.selectRemindedActivityIds(userId, activityIds);
        voList.forEach(vo -> vo.setReminded(remindedActivityIds.contains(vo.getId())));
    }

    // 按时间段分组
    Map<String, List<SeckillActivityVO>> timeSlotMap = new HashMap<>();
    for (SeckillActivityVO vo : voList) {
        String timeSlot = activities.stream()
                .filter(a -> a.getId().equals(vo.getId()))
                .findFirst()
                .map(a -> a.getStartTime().toLocalTime().toString().substring(0, 5))
                .orElse("00:00");
        timeSlotMap.computeIfAbsent(timeSlot, k -> new ArrayList<>()).add(vo);
    }

    List<TodaySeckillVO> result = new ArrayList<>();
    timeSlotMap.forEach((timeSlot, activityList) -> {
        result.add(TodaySeckillVO.builder()
                .timeSlot(timeSlot)
                .activities(activityList)
                .build());
    });

    result.sort(java.util.Comparator.comparing(TodaySeckillVO::getTimeSlot));
    return result;
}
```

```
- 快速失败，不占用系统资源
- 用户立即知道结果
- 避免排队等待的焦虑感
```

## 4. 技术难点与解决方案

### 4.1 超卖问题

**问题**: 多用户同时购买，库存变成负数

**解决方案**: Lua 脚本原子操作

```
┌─────────────────────────────────────────┐
│              超卖防护                    │
├─────────────────────────────────────────┤
│                                         │
│  方法1: 数据库乐观锁                     │
│  UPDATE stock SET count = count - 1     │
│  WHERE id = ? AND count > 0             │
│  → 性能差，数据库压力大                  │
│                                         │
│  方法2: Redis DECR                      │
│  DECR stock_key                         │
│  → 需要额外检查是否 < 0                  │
│                                         │
│  方法3: Lua 脚本 ✅                      │
│  if stock > 0 then DECR end             │
│  → 原子操作，高性能                      │
│                                         │
└─────────────────────────────────────────┘
```

### 4.2 竞态条件

**问题**: 库存预热和扣减之间存在时间窗口

**解决方案**: 预热+扣减合并为一个 Lua 脚本

```
❌ 两步操作的问题：

时间线: ─────────────────────────────────►

线程A: 检查库存不存在 → 预热库存(50) → [时间窗口] → 扣减(49)
线程B:                → 检查库存存在 → 扣减(49) → 扣减(48)
                                              ↑
                                        可能数据不一致

✅ Lua 脚本合并：

线程A: Lua 脚本原子执行 → 预热+扣减 → 返回 49
线程B: Lua 脚本原子执行 → 扣减 → 返回 48
```

### 4.3 最终一致性

**问题**: Redis 扣减成功但 MQ 发送失败

**解决方案**: 本地消息表 + 定时重试

```
场景：Redis 库存已扣减，用户获得购买资格，但 MQ 消息发送失败

处理流程：
1. 本地消息表记录状态为 PENDING
2. 定时任务扫描 PENDING 消息
3. 重新发送 MQ 消息
4. 发送成功后更新状态为 SENT
5. 超过最大重试次数标记为 FAILED，人工介入
```

### 4.4 缓存穿透

**问题**: 大量请求查询不存在的活动

**解决方案**: 多层校验

```java
// 1. 数据库查询前校验
SeckillActivity activity = activityMapper.selectById(activityId);
if (activity == null) {
    throw new BusinessException(5001, "秒杀活动不存在");

// 2. 可选：布隆过滤器（后续优化）

```

## 5. 面试题精选

### Q1: 秒杀系统如何防止超卖？

**答案**:

超卖是秒杀系统最核心的问题，常见解决方案有三种：

| 方案 | 实现方式 | 优点 | 缺点 |
|------|----------|------|------|
| **数据库乐观锁** | `UPDATE ... WHERE stock > 0` | 简单可靠 | 数据库压力大，QPS 低 |
| **Redis 分布式锁** | SETNX + EXPIRE | 实现简单 | 锁粒度大，并发度低 |
| **Lua 脚本原子操作** | `if stock > 0 then DECR end` | 高性能、原子性 | 需要 Redis 支持 |

**本项目采用 Lua 脚本方案**：

```lua
local stock = redis.call('GET', KEYS[1])
if stock and tonumber(stock) > 0 then
    return redis.call('DECR', KEYS[1])
end
return 0
```

**关键点**：Lua 脚本在 Redis 中原子执行，整个过程不会被其他命令打断。

### Q2: 为什么用 Redis 而不是数据库直接扣库存？

**答案**:

| 对比项 | MySQL | Redis |
|--------|-------|-------|
| **QPS** | ~1000 | ~100000 |
| **响应时间** | 10-50ms | <1ms |
| **并发模型** | 连接池限制 | 单线程事件循环 |
| **锁竞争** | 行锁竞争激烈 | 无锁（单线程） |

秒杀场景特点：**瞬时高并发、库存有限**

```
假设 10000 人同时秒杀 50 件商品：

MySQL 方案：
- 10000 个数据库连接请求
- 行锁竞争，串行执行
- 数据库可能崩溃

Redis 方案：
- 10000 个内存操作
- 单线程处理，无锁竞争
- 50 次成功，9950 次快速失败
- 数据库只处理 50 个订单创建请求
```

**结论**：Redis 作为前置缓存，承担高并发流量，保护后端数据库。

### Q3: 本地消息表是如何保证最终一致性的？

**答案**:

**问题背景**：Redis 扣减库存和 MQ 发送消息是两个独立操作，可能出现：
- Redis 成功，MQ 失败 → 用户扣了库存但没下单
- MQ 成功，Redis 失败 → 下了单但没扣库存

**本地消息表方案**：

```
┌─────────────────────────────────────────────────────────┐
│                   本地消息表流程                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 业务操作 + 消息记录 在同一数据库事务中               │
│     ┌─────────────────────────────────────────────┐     │
│     │ BEGIN TRANSACTION                            │     │
│     │ INSERT INTO local_message (status=PENDING)  │     │
│     │ COMMIT                                       │     │
│     └─────────────────────────────────────────────┘     │
│                                                         │
│  2. 尝试发送 MQ 消息                                     │
│     - 成功：更新状态为 SENT                              │
│     - 失败：保持 PENDING，等待重试                       │
│                                                         │
│  3. 定时任务扫描 PENDING 消息重试                        │
│     - 指数退避：1min → 2min → 5min → 10min → 30min     │
│     - 超过最大重试次数标记 FAILED                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**核心思想**：将分布式事务拆解为本地事务 + 可靠消息传递。
