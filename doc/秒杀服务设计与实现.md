## 1. 设计原理

### 1.1 秒杀业务特点

| 特点 | 说明 | 技术挑战 |
|------|------|----------|
| **瞬时高并发** | 活动开始瞬间流量激增 | 系统承载能力、数据库压力 |
| **库存有限** | 通常 10-50 件 | 防止超卖、数据一致性 |
| **时间限制** | 限时抢购 | 精准的时间控制 |
| **公平性** | 每人限购一件 | 用户去重 |
| **高可用** | 不能宕机 | 服务稳定性 |


```
                    ┌──────────────┐
                    │   客户端     │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  CDN/静态化  │  ← 第一层：静态资源加速
                    └──────┬───────┘

   │   Redis   │    │  Sentinel │    │   MQ      │  ← 第三层：缓存+限流+异步
```

| 层级 | 技术 | 原因 |

| **Lua 脚本** | 原子性保证 | 避免多命令间的竞态条件 |
| **RocketMQ** | 异步解耦 | 削峰填谷，保护订单服务 |
| **本地消息表** | 最终一致性 | 保证 Redis 扣减与 MQ 发送的一致性 |
| **Sentinel** | 限流保护 | 防止系统过载 |

## 3. 核心代码解析

### 3.1 Lua 脚本原子扣减

**文件**: [SeckillLuaScript.java](../backend/fnusale-marketing/src/main/java/com/fnusale/marketing/script/SeckillLuaScript.java)

#### 预热并扣减脚本（核心）

```java
public static final String WARM_UP_AND_DECREMENT_SCRIPT = """
    local stock = redis.call('GET', KEYS[1])
    if stock == false then
        -- 库存未预热，使用传入的值初始化
        if ARGV[1] and tonumber(ARGV[1]) > 0 then
            redis.call('SET', KEYS[1], ARGV[1])
            if ARGV[2] and tonumber(ARGV[2]) > 0 then
                redis.call('EXPIRE', KEYS[1], ARGV[2])
            end
            stock = ARGV[1]
        else
            return -1
        end
    end

    local remain = tonumber(stock)
    if remain <= 0 then
        return 0
    end

    return redis.call('DECR', KEYS[1])
    """;
```

**设计要点**:

| 要点 | 说明 |
|------|------|
| **原子性** | 整个脚本在 Redis 中原子执行，不会被其他命令打断 |
| **预热+扣减合并** | 解决了两步操作间的竞态条件 |
| **返回值明确** | 正数=剩余库存，0=库存不足，-1=参数错误 |

**为什么用 Lua 脚本？**

```
❌ 不使用 Lua 的问题：

线程A: GET stock → 返回 1
线程B: GET stock → 返回 1
线程A: DECR stock → 返回 0
线程B: DECR stock → 返回 -1  ← 超卖！

✅ 使用 Lua 的解决方案：

线程A: 执行 Lua 脚本 → 原子操作 → 返回成功
线程B: 执行 Lua 脚本 → 原子操作 → 返回库存不足
```

### 3.2 秒杀核心方法

**文件**: [SeckillServiceImpl.java](../backend/fnusale-marketing/src/main/java/com/fnusale/marketing/service/impl/SeckillServiceImpl.java)

```java
@Override
@Transactional(rollbackFor = Exception.class)
public Long joinSeckill(Long userId, Long activityId) {
    // 1. 查询活动信息
    SeckillActivity activity = activityMapper.selectById(activityId);
    if (activity == null) {
        throw new BusinessException(5001, "秒杀活动不存在");
    }

    // 2. 检查活动状态
    if (MarketingConstants.SECKILL_STATUS_NOT_START.equals(activity.getActivityStatus())) {
        throw new BusinessException(5002, "秒杀活动尚未开始");
    }
    if (MarketingConstants.SECKILL_STATUS_END.equals(activity.getActivityStatus())) {
        throw new BusinessException(5003, "秒杀活动已结束");
    }

    // 3. 检查用户是否已参与（Redis Set）
    String boughtKey = MarketingConstants.SECKILL_USER_BOUGHT_PREFIX + activityId;
    Boolean hasBought = redisTemplate.opsForSet().isMember(boughtKey, userId.toString());
    if (Boolean.TRUE.equals(hasBought)) {
        throw new BusinessException(5005, "您已参与过该秒杀");
    }

    // 4. Lua 脚本原子预热并扣减库存
    String stockKey = MarketingConstants.SECKILL_STOCK_KEY_PREFIX + activityId;
    long expireSeconds = Duration.between(LocalDateTime.now(), activity.getEndTime()).getSeconds();

    Long result = redisTemplate.execute(
            warmUpAndDecrementScript,
            Collections.singletonList(stockKey),
            activity.getRemainStock().toString(),
            String.valueOf(expireSeconds)
    );

    if (result == null || result == 0) {
        throw new BusinessException(5004, "秒杀库存不足");
    }

    // 5. 标记用户已购买
    redisTemplate.opsForSet().add(boughtKey, userId.toString());
    redisTemplate.expire(boughtKey, expireSeconds, TimeUnit.SECONDS);

    // 6. 本地消息表 + MQ 发送（确保最终一致性）
    // ... 省略消息发送代码

    return null; // 订单异步创建中
}
```

## 4. 技术难点与解决方案

### 4.1 超卖问题

**问题**: 多用户同时购买，库存变成负数

**解决方案**: Lua 脚本原子操作

```
┌─────────────────────────────────────────┐
│              超卖防护                    │
├─────────────────────────────────────────┤
│                                         │
│  方法1: 数据库乐观锁                     │
│  UPDATE stock SET count = count - 1     │
│  WHERE id = ? AND count > 0             │
│  → 性能差，数据库压力大                  │
│                                         │
│  方法2: Redis DECR                      │
│  DECR stock_key                         │
│  → 需要额外检查是否 < 0                  │
│                                         │
│  方法3: Lua 脚本 ✅                      │
│  if stock > 0 then DECR end             │
│  → 原子操作，高性能                      │
│                                         │
└─────────────────────────────────────────┘
```

### 4.2 竞态条件

**问题**: 库存预热和扣减之间存在时间窗口

**解决方案**: 预热+扣减合并为一个 Lua 脚本

```
❌ 两步操作的问题：

时间线: ─────────────────────────────────►

线程A: 检查库存不存在 → 预热库存(50) → [时间窗口] → 扣减(49)
线程B:                → 检查库存存在 → 扣减(49) → 扣减(48)
                                              ↑
                                        可能数据不一致

✅ Lua 脚本合并：

线程A: Lua 脚本原子执行 → 预热+扣减 → 返回 49
线程B: Lua 脚本原子执行 → 扣减 → 返回 48
```

### 4.3 最终一致性

**问题**: Redis 扣减成功但 MQ 发送失败

**解决方案**: 本地消息表 + 定时重试

```
场景：Redis 库存已扣减，用户获得购买资格，但 MQ 消息发送失败

处理流程：
1. 本地消息表记录状态为 PENDING
2. 定时任务扫描 PENDING 消息
3. 重新发送 MQ 消息
4. 发送成功后更新状态为 SENT
5. 超过最大重试次数标记为 FAILED，人工介入
```

### 4.4 缓存穿透

**问题**: 大量请求查询不存在的活动

**解决方案**: 多层校验

```java
// 1. 数据库查询前校验
SeckillActivity activity = activityMapper.selectById(activityId);
if (activity == null) {
    throw new BusinessException(5001, "秒杀活动不存在");
}

// 2. 可选：布隆过滤器（后续优化）
// 在 Redis 中预存所有有效活动 ID
// 请求先经过布隆过滤器判断
```

## 5. 面试题精选

### Q1: 秒杀系统如何防止超卖？

**答案**:

超卖是秒杀系统最核心的问题，常见解决方案有三种：

| 方案 | 实现方式 | 优点 | 缺点 |
|------|----------|------|------|
| **数据库乐观锁** | `UPDATE ... WHERE stock > 0` | 简单可靠 | 数据库压力大，QPS 低 |
| **Redis 分布式锁** | SETNX + EXPIRE | 实现简单 | 锁粒度大，并发度低 |
| **Lua 脚本原子操作** | `if stock > 0 then DECR end` | 高性能、原子性 | 需要 Redis 支持 |

**本项目采用 Lua 脚本方案**：

```lua
local stock = redis.call('GET', KEYS[1])
if stock and tonumber(stock) > 0 then
    return redis.call('DECR', KEYS[1])
end
return 0
```

**关键点**：Lua 脚本在 Redis 中原子执行，整个过程不会被其他命令打断。

---

### Q2: 为什么用 Redis 而不是数据库直接扣库存？

**答案**:

| 对比项 | MySQL | Redis |
|--------|-------|-------|
| **QPS** | ~1000 | ~100000 |
| **响应时间** | 10-50ms | <1ms |
| **并发模型** | 连接池限制 | 单线程事件循环 |
| **锁竞争** | 行锁竞争激烈 | 无锁（单线程） |

秒杀场景特点：**瞬时高并发、库存有限**

```
假设 10000 人同时秒杀 50 件商品：

MySQL 方案：
- 10000 个数据库连接请求
- 行锁竞争，串行执行
- 数据库可能崩溃

Redis 方案：
- 10000 个内存操作
- 单线程处理，无锁竞争
- 50 次成功，9950 次快速失败
- 数据库只处理 50 个订单创建请求
```

**结论**：Redis 作为前置缓存，承担高并发流量，保护后端数据库。

### Q3: 本地消息表是如何保证最终一致性的？

**答案**:

**问题背景**：Redis 扣减库存和 MQ 发送消息是两个独立操作，可能出现：
- Redis 成功，MQ 失败 → 用户扣了库存但没下单
- MQ 成功，Redis 失败 → 下了单但没扣库存

**本地消息表方案**：

```
┌─────────────────────────────────────────────────────────┐
│                   本地消息表流程                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 业务操作 + 消息记录 在同一数据库事务中               │
│     ┌─────────────────────────────────────────────┐     │
│     │ BEGIN TRANSACTION                            │     │
│     │ INSERT INTO local_message (status=PENDING)  │     │
│     │ COMMIT                                       │     │
│     └─────────────────────────────────────────────┘     │
│                                                         │
│  2. 尝试发送 MQ 消息                                     │
│     - 成功：更新状态为 SENT                              │
│     - 失败：保持 PENDING，等待重试                       │
│                                                         │
│  3. 定时任务扫描 PENDING 消息重试                        │
│     - 指数退避：1min → 2min → 5min → 10min → 30min     │
│     - 超过最大重试次数标记 FAILED                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**核心思想**：将分布式事务拆解为本地事务 + 可靠消息传递。

### Q6: 库存预热是什么？为什么要预热？

**答案**:

**预热**：在活动开始前，将库存数据提前加载到 Redis 中。

**不预热的问题**：

```
活动开始瞬间 10000 请求涌入：

第一个请求：
  Redis GET → 返回 null（不存在）
  查询数据库 → 得到库存 50
  写入 Redis
  扣减库存

期间 9999 个请求也都在查数据库 → 数据库崩溃
```

**预热后的流程**：

```
活动开始前 5 分钟：
  定时任务将库存写入 Redis
  SET seckill:stock:1 50

活动开始：
  所有请求直接操作 Redis
  无数据库压力
```

**本项目预热机制**：

```java
// 创建活动时调度预热
private void scheduleWarmUp(SeckillActivity activity) {

}
```

### Q8: 秒杀结果如何查询？

**答案**:

由于订单是异步创建的，用户秒杀成功后不能立即看到订单，需要轮询查询。

**查询流程**：

```java
public SeckillResultVO getSeckillResult(Long userId, Long activityId) {
    // 1. 检查用户是否参与了秒杀
    String boughtKey = "seckill:user:bought:" + activityId;
    Boolean hasBought = redisTemplate.opsForSet().isMember(boughtKey, userId.toString());

    if (Boolean.TRUE.equals(hasBought)) {
        // 2. 查询订单服务获取订单状态
        var result = orderClient.getSeckillOrderStatus(userId, activityId);
        if (result.isSuccess() && result.getData() != null) {
            OrderVO order = result.getData();
            return SeckillResultVO.builder()
                    .success(true)
                    .orderId(order.getId())
                    .orderStatus(order.getOrderStatus())
                    .build();
        }
        // 3. 订单还在创建中
        return SeckillResultVO.success("订单创建中");
    }

    return SeckillResultVO.fail("您未参与该秒杀活动");
}
```

**前端轮询策略**：

```javascript
// 轮询间隔：500ms → 1s → 2s（指数退避）
async function checkResult(activityId) {
    const result = await api.getSeckillResult(activityId);
    if (result.success && result.orderId) {
        // 跳转到订单详情
        location.href = '/order/' + result.orderId;
    } else if (result.message === '订单创建中') {
        // 继续轮询
        setTimeout(() => checkResult(activityId), 1000);
    } else {
        // 秒杀失败
        showError(result.message);
    }
}
```
