# 校园二手交易平台 - 数据库结构设计（贴合业务需求，无具体SQL）

## 设计原则

1. **贴合业务场景**：字段完全匹配前文功能分析，无冗余、无遗漏；
2. **轻量实用**：拒绝复杂表结构，适配校园规模数据量；
3. **数据安全**：敏感字段（学号、手机号）脱敏存储，核心数据加逻辑删除；
4. **关联清晰**：通过外键/唯一键保证表间关联，核心字段加索引提升查询效率。

## 一、用户模块表设计

### 1. 用户基础表（t_user）

| 字段名              | 字段说明                                                       | 核心设计点                                |
| ------------------- | -------------------------------------------------------------- | ----------------------------------------- |
| id                  | 主键（自增/雪花ID）                                            | 唯一标识用户                              |
| student_teacher_id  | 学号/工号（唯一）                                              | 脱敏存储（仅显示后4位），校园身份核心标识 |
| username            | 用户名（昵称）                                                 | 非唯一，支持修改                          |
| phone               | 手机号                                                         | 脱敏存储（138\*\*\*\*8000），加索引       |
| campus_email        | 校园邮箱                                                       | 唯一，用于身份核验（可选）                |
| password            | 密码                                                           | 加密存储（BCrypt）                        |
| identity_type       | 身份类型（枚举：STUDENT/TEACHER）                              | 区分学生/教职工                           |
| auth_status         | 认证状态（枚举：UNAUTH/UNDER_REVIEW/AUTH_SUCCESS/AUTH_FAILED） | 未认证/审核中/认证成功/认证失败           |
| auth_image_url      | 校园卡/学生证审核图片地址（OSS）                               | 存储审核凭证                              |
| auth_result_remark  | 认证审核备注                                                   | 认证失败时记录原因（如“校园卡信息模糊”）  |
| location_permission | 定位权限状态（枚举：ALLOW/DENY）                               | 记录用户是否授权定位                      |
| credit_score        | 信誉分（默认100，违规扣减）                                    | 低信誉（<60）限制发布/秒杀                |
| create_time         | 创建时间                                                       | 通用字段                                  |
| update_time         | 更新时间                                                       | 通用字段                                  |
| is_deleted          | 逻辑删除标记（0/1）                                            | 避免物理删除数据                          |

## 总结

### 核心设计亮点

1. **贴合业务**：所有表字段完全匹配前文功能需求（如校园认证审核、定位经纬度、秒杀库存、IM消息类型等）；
2. **轻量高效**：无复杂关联和冗余字段，核心检索字段（用户ID、商品ID、订单编号）加索引，适配校园规模查询；
3. **安全合规**：敏感字段脱敏、核心数据逻辑删除、操作日志可追溯；
4. **易扩展**：预留扩展字段（如商品扩展属性、用户扩展信息），支持后续功能迭代。
