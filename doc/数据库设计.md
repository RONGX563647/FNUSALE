## 二、数据库整体设计概述

### 一、业务背景与设计目标

#### 1.1 业务背景

校园二手交易平台是面向高校师生的C2C二手商品交易平台，核心业务场景包括：

- 用户注册与校园身份认证
- 商品发布与浏览（支持AI自动分类）
- 即时沟通（IM）
- 交易下单与自提
- 营销活动（优惠券、秒杀）
- 用户评价与信誉体系
- 运营管理与数据分析

#### 1.2 核心业务场景

| 业务场景 | 峰值流量 | 数据量级预估 | 年增量 | 存储周期 |
|----------|----------|--------------|--------|----------|
| 用户注册/登录 | 500 QPS | 10万用户 | 5万/年 | 永久 |
| 商品发布 | 200 QPS | 5万商品 | 3万/年 | 3年 |
| 商品浏览 | 2000 QPS | 5万商品 | - | 3年 |
| IM消息 | 1000 QPS | 1000万条/月 | 1200万/年 | 6个月 |
| 交易下单 | 300 QPS | 20万订单 | 15万/年 | 5年 |
| 秒杀活动 | 5000 QPS | 100次/月 | 1200次/年 | 1年 |

#### 1.3 SLA承诺

| 指标 | 承诺值 | 说明 |
|------|--------|------|
| 系统可用性 | 99.9% | 年度停机时间不超过8.76小时 |
| RTO | 30分钟 | 故障恢复时间不超过30分钟 |
| RPO | 5分钟 | 数据丢失不超过5分钟 |
| 响应时间 | < 200ms | 95%请求响应时间小于200ms |
| 并发用户 | 10000 | 支持10000并发用户 |

#### 1.4 核心设计原则

| 原则 | 具体要求 |
|------|----------|
| 范式要求 | OLTP场景采用3NF，避免数据冗余，保证数据一致性 |
| 读写分离 | 采用主从架构，读写分离，提升查询性能 |
| 分库分表 | 暂不分库分表，单表数据量超过1000万时考虑分表 |
| 冷热分离 | IM消息、日志等数据定期归档，热数据保留6个月 |
| 数据一致性 | 采用最终一致性，通过本地消息表保障分布式事务 |
| 安全合规 | 敏感数据加密存储，符合个保法、等保2.0要求 |

### 二、数据库选型与环境规划

#### 2.1 数据库选型

| 选型项 | 选型内容 | 选型理由 |
|--------|----------|----------|
| 数据库类型 | OLTP | 面向交易型业务场景，高并发读写 |
| 产品版本 | MySQL 8.0.32 | 成熟稳定，社区活跃，支持JSON、窗口函数等特性 |
| 集群架构 | 主从复制 + MGR | 主从复制提供读写分离，MGR提供高可用 |
| 存储引擎 | InnoDB | 支持事务、行锁、外键，适合OLTP场景 |
| 字符集 | utf8mb4 | 支持emoji等4字节字符 |

#### 2.2 环境划分

| 生产环境 | 正式运行 | DBA完全权限，应用只读/读写权限 | 全加密 | - |

### 四、全局设计规范

#### 4.1 命名规范

| 对象类型 | 命名规则 | 示例 | 说明 |
|----------|----------|------|------|
| 数据库 | 小写+下划线 | fnusale | 禁止使用SQL关键字 |
| Schema | 小写+下划线 | fnusale | 与数据库名一致 |
| 表名 | 小写+下划线，业务域前缀 | t_user, t_product | t_前缀标识表 |
| 字段名 | 小写+下划线 | user_id, create_time | 禁止使用SQL关键字 |
| 索引名 | idx_表名_字段名 | idx_user_phone | 普通索引 |
| 唯一索引 | uk_表名_字段名 | uk_user_student_id | 唯一索引 |
| 主键索引 | pk_表名 | pk_user | 主键索引 |
| 外键 | fk_表名_字段名 | fk_order_user_id | 外键约束 |
| 约束名 | chk_表名_字段名 | chk_user_credit_score | CHECK约束 |

#### 4.2 范式规范

| 范式等级 | 适用场景 | 说明 |
|----------|----------|------|
| 3NF | OLTP场景 | 消除传递依赖，保证数据一致性 |
| 2NF | 分析场景 | 允许适度反范式，提升查询性能 |
| 例外场景 | - | 统计表、缓存表允许反范式 |

#### 4.3 字段设计规范

| 规则类型 | 具体要求 | 示例 |
|----------|----------|------|
| 数据类型选型 | 金额用DECIMAL，禁止VARCHAR(255)一刀切 | DECIMAL(10,2) |
| 必填规则 | 主键、外键、核心业务字段非空 | NOT NULL |
| 默认值规范 | 数值类型默认0，字符串默认空字符串 | DEFAULT 0, DEFAULT '' |
| 时间字段 | 统一使用DATETIME或TIMESTAMP | DATETIME, TIMESTAMP |
| 枚举字段 | 使用TINYINT存储，注释说明枚举值 | TINYINT COMMENT '0:未认证 1:已认证' |
| 状态字段 | 使用TINYINT，0表示禁用/否，1表示启用/是 | TINYINT DEFAULT 0 |

#### 4.4 索引设计规范

| 规则类型 | 具体要求 | 示例 |
|----------|----------|------|
| 主键设计 | 禁止业务字段做主键，推荐自增ID/雪花ID | BIGINT AUTO_INCREMENT |
| 联合索引顺序 | 区分度高的字段在前，等值查询在前 | idx_user_phone_status(phone, status) |
| 唯一索引 | 业务唯一字段必须加唯一索引 | uk_user_student_id(student_id) |
| 索引禁用场景 | 区分度低的字段、频繁更新的字段不建索引 | 状态字段、性别字段 |
| 索引维护 | 定期分析索引使用情况，删除无用索引 | - |

#### 4.5 约束设计规范

| 约束类型 | 使用规则 | 风险控制 |
|----------|----------|----------|
| 主键约束 | 所有表必须有主键 | 使用自增ID或雪花ID |
| 外键约束 | 分布式场景使用逻辑外键，单机场景使用物理外键 | 物理外键影响性能，慎用 |
| 非空约束 | 核心业务字段必须非空 | 保证数据完整性 |
| CHECK约束 | 数据校验规则 | MySQL 8.0.16+支持 |

## 三、详细数据模型设计

### 一、分层数据模型设计

#### 1.1 概念数据模型（CDM）

**核心业务实体**：

- 用户（User）：学生、教职工
- 商品（Product）：二手商品
- 订单（Order）：交易订单
- 消息（Message）：IM即时消息
- 优惠券（Coupon）：营销优惠券
- 活动（Activity）：秒杀活动
- 评价（Evaluation）：用户评价

**实体关系**：

```
User 1:N Product      (一个用户可以发布多个商品)
User 1:N Order        (一个用户可以创建多个订单)
Product 1:N Order      (一个商品可以被多次交易)
User M:N Message      (用户之间可以互相发送消息)
User 1:N Evaluation    (一个用户可以收到多个评价)
Coupon 1:N UserCoupon (一个优惠券可以被多个用户领取)
Activity 1:N Product  (一个活动可以关联多个商品)
```

#### 1.2 逻辑数据模型（LDM）

**用户模块**：

- 用户基础信息（id, student_id, username, phone, email, password, identity_type, auth_status, credit_score, ...）
- 用户地址（id, user_id, address_type, pick_point_id, custom_address, ...）
- 用户积分（id, user_id, total_points, available_points, ...）
- 用户评价分（id, user_id, overall_rating, rating_level, ...）

**商品模块**：

- 商品品类（id, category_name, parent_category_id, ai_mapping_value, ...）
- 商品基础信息（id, user_id, category_id, product_name, new_degree, price, ...）
- 商品图片（id, product_id, image_url, is_main_image, sort, ...）

**交易模块**：

- 订单（id, order_no, user_id, product_id, actual_pay_amount, order_status, ...）
- 订单评价（id, order_id, evaluator_id, evaluated_id, score, ...）
- 交易纠纷（id, order_id, initiator_id, accused_id, dispute_type, ...）

## 四、数据安全与合规设计

| 级别 | 定义 | 示例字段 | 保护要求 |

|--------|----------|----------|----------|

| 100万-1000万行 | Online DDL | 业务低峰期 | 中影响 |
| > 1000万行 | pt-online-schema-change | 业务低峰期 | 高影响 |
