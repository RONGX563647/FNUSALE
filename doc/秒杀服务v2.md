# 秒杀服务 v2 问题分析与初步优化文档

## 概述

本文档记录了秒杀服务从 v1 到 v2 的问题发现、分析和初步优化过程。v2 版本主要发现了 v1 版本中存在的严重问题，并尝试了初步的解决方案。

---

## 一、问题发现

### 1.1 问题背景

v1 版本上线后，通过线上监控和用户反馈，发现了以下问题：

| 问题 | 发现方式 | 影响程度 |
|------|----------|----------|
| **Lua 返回值语义混淆** | 用户反馈"明明抢到了却提示失败" | 严重 |
| **Redis 与数据库不一致** | 数据对账发现差异 | 严重 |
| **回滚操作非原子** | 代码审查发现 | 中等 |

---

## 二、问题详细分析

### 2.1 P0 级别问题（严重）

#### 问题1：Lua 返回值语义混淆（导致少卖）

**问题描述**：
Lua 脚本 `SECKILL_ATOMIC_SCRIPT` 中，当库存为 1 时用户抢购成功，`DECR` 操作后返回 `0`。但 Java 代码将 `result == 0` 误判为"库存不足"。

**问题代码**：

```java
// v1 版本错误处理
if (result == null || result == 0) {
    throw new BusinessException(5004, "秒杀库存不足");
}
```

**Lua 脚本执行流程**（库存=1 时）：

```lua
local remain = tonumber(stock)  -- remain = 1
if remain <= 0 then return 0 end  -- 1 <= 0? false, 继续

local queueNumber = redis.call('DECR', KEYS[1])  -- 1 - 1 = 0
if queueNumber < 0 then ... end  -- 0 < 0? false, 继续

redis.call('SADD', KEYS[2], ARGV[1])  -- 标记用户已购买
return queueNumber  -- 返回 0
```

**严重后果**：

- 最后一个库存被成功扣减
- 用户被 Redis 标记为"已购买"
- **但 Java 抛出异常，用户被告知失败**
- 导致**少卖（underselling）**问题

---

#### 🔬 问题1技术原理分析

**1. 问题具体表现**

当库存为 **1** 时，用户抢购成功，但系统却告知用户"库存不足"，导致**少卖（underselling）**问题。

**2. 根本原因分析**

**Lua 脚本返回值与 Java 处理逻辑的语义不匹配**：

```
Lua 脚本实际行为：
- 库存=1 时，DECR 返回 0（表示扣减后剩余 0）
- 这代表"成功抢到最后一件"

Java 代码错误理解：
- result == 0 被误判为"库存不足"
- 导致成功抢购的用户被拒绝
```

**3. 技术原理详解**

**Redis DECR 命令的工作原理**：

```
DECR 命令执行过程：
1. 读取当前值：stock = 1
2. 执行减法：1 - 1 = 0
3. 写入新值：stock = 0
4. 返回新值：return 0

关键点：DECR 返回的是操作后的值，不是操作是否成功的布尔值
```

**问题发生的完整时序**：

```
时间线：
T1: 用户发起秒杀请求
T2: Lua 脚本执行成功
    - Redis 库存从 1 变为 0
    - 用户被添加到"已购买集合"
    - 返回值 = 0
T3: Java 代码判断
    - if (result == 0) → true
    - throw new BusinessException("秒杀库存不足")
T4: 用户收到"库存不足"错误

结果：
- Redis 状态：库存=0，用户已标记购买
- 数据库状态：无订单记录
- 用户感知：秒杀失败
- 实际情况：用户本应成功抢到最后一件
```

---

#### 问题2：Redis 与数据库不一致

**问题描述**：
Redis 操作成功后，如果数据库操作失败，没有回滚 Redis 的机制。

**问题代码**：

```java
// v1 版本
// Redis 操作成功
Long result = redisTemplate.execute(...);

try {
    // 数据库操作可能失败
    localMessageMapper.insert(localMessage);
} catch (Exception e) {
    // 没有 Redis 回滚！
    throw new BusinessException(500, "系统繁忙");
}
```

**严重后果**：

- Redis 库存已扣减
- 用户已被标记购买
- 但数据库没有记录
- 用户无法查询到订单

---

#### 🔬 问题2技术原理分析

**1. 问题具体表现**

Redis 库存扣减成功后，如果数据库操作失败，没有回滚 Redis 的机制，导致：
- Redis 库存已扣减
- 用户已被标记购买
- 但数据库没有记录
- 用户无法查询到订单

**2. 根本原因分析**

**Redis 操作在数据库事务范围外，数据库回滚不会自动回滚 Redis**。

**3. 技术原理详解**

**Spring 事务与 Redis 的隔离性**：

```
Spring @Transactional 事务范围：
┌─────────────────────────────────────────┐
│ BEGIN TRANSACTION                        │
│   └─ 数据库操作（受事务保护）             │
│   └─ 异常时自动回滚                       │
│ COMMIT / ROLLBACK                        │
└─────────────────────────────────────────┘

Redis 操作：
┌─────────────────────────────────────────┐
│ Redis 操作（事务范围外）                  │
│   └─ 不受 Spring 事务管理                │
│   └─ 数据库回滚时不会自动回滚             │
└─────────────────────────────────────────┘
```

**问题场景时序图**：

```
时间线：
T1: Redis Lua 脚本执行成功
    - 库存扣减：50 → 49
    - 用户标记：已购买

T2: 开始数据库事务
    - 尝试插入本地消息表

T3: 数据库操作失败
    - 原因：连接超时/约束违反/磁盘满等
    - Spring 事务回滚

T4: Redis 状态未恢复
    - 库存仍为 49（少 1）
    - 用户仍标记为已购买

T5: 用户查询订单
    - 数据库无记录
    - Redis 显示已购买
    - 用户困惑：明明显示已购买，为什么没订单？
```

**为什么 Redis 不受 Spring 事务管理？**

```
Spring 事务管理原理：
- 基于 AOP 代理
- 只管理实现了 TransactionAware 接口的资源
- 主要是 JDBC Connection、JPA EntityManager

Redis 连接：
- RedisTemplate 使用 RedisConnection
- 不实现 TransactionAware 接口
- 不参与 Spring 声明式事务
```

### 2.2 P1 级别问题

#### 问题3：getSeckillResult 传参错误

**问题描述**：
调用 `SeckillResultVO.success()` 时传入时间戳，而非订单ID。

**问题代码**：

```java
return SeckillResultVO.success(System.currentTimeMillis());  // 时间戳当 orderId？
```

**影响**：API 语义不清晰，前端可能误解返回值。

#### 问题4：rollbackRedis 非原子操作

**问题描述**：
回滚 Redis 时使用两个独立的 Redis 命令，非原子操作。

**问题代码**：

```java
// v1 版本回滚（如果有的话）
// 恢复库存
redisTemplate.opsForValue().increment(stockKey);
// 移除购买标记
redisTemplate.opsForSet().remove(boughtKey, userId.toString());
```

**风险**：
- increment 成功，remove 失败 → 库存恢复，但用户仍显示"已购买"
- remove 成功，increment 失败 → 用户标记清除，但库存未恢复

---

#### 🔬 问题4技术原理分析

**非原子操作的风险场景**：

```
原实现：
redisTemplate.opsForValue().increment(stockKey);  // 操作1
redisTemplate.opsForSet().remove(boughtKey, userId.toString());  // 操作2

风险场景1：操作1成功，操作2失败
T1: increment 成功 → 库存恢复
T2: remove 失败 → 用户标记未清除
结果：库存正确，但用户仍显示"已购买"

风险场景2：操作2成功，操作1失败
T1: remove 成功 → 用户标记清除
T2: increment 失败 → 库存未恢复
结果：用户可重新购买，但库存少1

风险场景3：并发干扰
线程A: increment → 库存 49 → 50
线程B: decrement → 库存 50 → 49  ← 在两次操作之间插入
线程A: remove → 用户标记清除
结果：库存状态混乱
```

## 三、v2 版本初步解决方案

### 3.1 尝试修复 Lua 返回值语义

**方案**：尝试区分返回值含义，但仍有问题。

**修改后的 Lua 脚本（v2 尝试）**：

```lua
-- v2 版本尝试修复（仍有问题）
-- 返回值: 0=库存不足, >0=成功

-- 问题：仍然用 0 表示失败，与 DECR 返回 0 冲突
if remain <= 0 then
    return 0  -- 问题：与抢到最后一件的返回值冲突
end

if queueNumber < 0 then
    redis.call('INCR', KEYS[1])
    return 0  -- 问题：同上
end

return queueNumber  -- 可能返回 0（抢到最后一件）
```

**问题**：v2 版本的修复不彻底，仍然存在返回值语义混淆的问题。

---

### 3.2 尝试添加 Redis 回滚

**方案**：在数据库操作失败时尝试回滚 Redis。

**Java 回滚方法（v2 尝试）**：

```java
// v2 版本尝试（非原子）
private void rollbackRedis(Long activityId, Long userId) {
    try {
        String stockKey = MarketingConstants.SECKILL_STOCK_KEY_PREFIX + activityId;
        String boughtKey = MarketingConstants.SECKILL_USER_BOUGHT_PREFIX + activityId;

        // 问题：两个独立操作，非原子
        redisTemplate.opsForValue().increment(stockKey);
        redisTemplate.opsForSet().remove(boughtKey, userId.toString());
        
        log.warn("Redis 回滚: activityId={}, userId={}", activityId, userId);
    } catch (Exception e) {
        log.error("Redis 回滚失败: activityId={}, userId={}", activityId, userId, e);
    }
}
```

**问题**：回滚操作非原子，可能被打断。

---

### 3.3 修复 getSeckillResult

**方案**：使用 `SeckillResultVO.pending()` 方法返回正确的状态。

**修改前**：

```java
return SeckillResultVO.success(System.currentTimeMillis());
```

**修改后**：

```java
return SeckillResultVO.pending(null, activityId);
```

## 五、面试题精选（基于 v2 问题）

### Q1: v2 版本为什么没有完全解决 Lua 返回值语义混淆问题？

**答案**:

**问题分析**：

v2 版本尝试修复返回值语义，但仍使用 0 表示"库存不足"，这与 DECR 返回 0（表示抢到最后一件）产生冲突。

**根本原因**：

```
v2 版本的返回值设计：
- 0 表示库存不足
- >0 表示成功

问题：
- 库存=1 时，DECR 返回 0
- 这个 0 表示"成功抢到最后一件"
- 但代码判断 result == 0 为失败

正确做法（v3 版本）：
- >= 0 表示成功（排队号）
- < 0 表示失败（不同负数代表不同错误）
```

---

### Q2: v2 版本的回滚机制有什么问题？

**答案**:

**问题分析**：

v2 版本的回滚使用两个独立的 Redis 命令：

```java
redisTemplate.opsForValue().increment(stockKey);    // 命令1
redisTemplate.opsForSet().remove(boughtKey, userId); // 命令2
```

**风险场景**：

```
风险1：命令1成功，命令2失败
- 库存恢复，但用户标记未清除
- 用户仍显示"已购买"，无法再次购买

风险2：命令2成功，命令1失败
- 用户标记清除，但库存未恢复
- 用户可重新购买，但库存少 1

风险3：并发干扰
- 两个命令之间可能被其他操作打断
- 导致数据状态混乱
```

**正确做法（v3 版本）**：

使用 Lua 脚本保证回滚操作的原子性。

---

### Q3: 如何设计 Lua 脚本的返回值，避免语义混淆？

**答案**:

**设计原则**：

```
┌─────────────────────────────────────────────────────────┐
│              Lua 脚本返回值设计原则                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 语义清晰原则                                        │
│     - 同一个值不要表示多种含义                          │
│     - ❌ 0 同时表示"成功"和"失败"                       │
│     - ✅ >=0 成功，<0 失败                              │
│                                                         │
│  2. 正数成功原则                                        │
│     - 正数/0 表示成功，可携带业务数据                   │
│     - 负数表示失败，不同负数区分错误类型                │
│                                                         │
│  3. 文档同步原则                                        │
│     - Lua 脚本注释与 Java 代码注释保持一致              │
│     - 返回值定义写入常量类                              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**v3 版本返回值定义**：

| 返回值 | 含义 | 说明 |
|--------|------|------|
| `>= 0` | 秒杀成功 | 返回排队号（0 表示抢到最后一件） |
| `-1` | 参数错误 | 预热库存值无效 |
| `-2` | 用户已购买 | 防止重复购买 |
| `-3` | 预热值无效 | 库存未预热且预热值无效 |
| `-4` | 库存不足 | 库存为 0 或并发竞态失败 |

### Q4: 为什么 Redis 不受 Spring 事务管理？

**答案**:

**Spring 事务管理原理**：

```
Spring 事务管理机制：
1. 基于 AOP 代理
2. 只管理实现了 TransactionAware 接口的资源
3. 主要是 JDBC Connection、JPA EntityManager

工作流程：
1. 方法开始时，获取数据库连接
2. 设置 autoCommit = false
3. 执行业务操作
4. 方法正常结束，提交事务
5. 方法抛出异常，回滚事务
```

**Redis 连接特性**：

```
RedisTemplate 特性：
1. 使用 RedisConnection
2. 不实现 TransactionAware 接口
3. 不参与 Spring 声明式事务

结果：
- Redis 操作独立于数据库事务
- 数据库回滚不会影响 Redis
- 需要手动实现补偿机制
```

### Q5: v1 到 v2 的演进过程中，最大的教训是什么？

**答案**:

**教训1：返回值语义必须清晰**

```
问题：
- 使用 0 同时表示"成功"和"失败"
- 导致边界条件（库存=1）处理错误

教训：
- 返回值语义必须明确
- 正数/零表示成功，负数表示失败
- 边界条件必须测试覆盖
```

**教训2：回滚操作必须原子**

```
问题：
- 使用两个独立命令回滚
- 可能被打断，导致数据不一致

教训：
- 多个 Redis 操作必须用 Lua 脚本保证原子性
- 回滚操作本身也要保证原子性
```

**教训3：Redis 与数据库一致性需要补偿机制**

```
问题：
- Redis 不受 Spring 事务管理
- 数据库回滚不会自动回滚 Redis

教训：
- 必须实现补偿机制
- 使用本地消息表保证最终一致性
- 定时对账发现并修复不一致
```
