# 秒杀服务 v4 优化文档

## 概述

本文档记录了秒杀服务从 v3 到 v4 的优化过程，主要解决了 v3 版本遗留的数据一致性、内存泄漏、用户体验、防刷机制等问题。

---

## 一、v3 遗留问题分析

### 1.1 P0 级别问题（严重）

#### 问题1：Redis库存与数据库库存不一致

**问题描述**：
- Lua脚本扣减的是 **Redis 中的库存**
- 但 **数据库中的 `remain_stock` 没有同步扣减**
- 活动结束后，数据库的 `remain_stock` 仍然是初始值

**影响**：
- 数据不一致，影响数据统计和报表
- 如果 Redis 数据丢失，会导致超卖

#### 问题2：活动结束后Redis数据未清理

**问题描述**：
活动结束后，Redis 中的库存、用户购买标记、时间戳key没有清理，会持续占用内存。

**影响**：
- Redis 内存泄漏
- 长期运行后可能导致 Redis 内存不足

#### 问题3：订单创建失败时缺少回滚机制

**问题描述**：
如果 `SeckillOrderConsumer` 创建订单失败，Redis 库存已经扣减，但没有回滚。

**影响**：
- 用户扣减了库存但没创建订单
- 用户会一直看到 "订单创建中" 的状态

---

### 1.2 P1 级别问题

#### 问题4：秒杀结果查询缺少超时处理

**问题描述**：
如果订单创建一直失败，用户查询结果会一直返回 `pending` 状态，用户无限等待。

#### 问题5：缺少防刷机制

**问题描述**：
当前只有用户购买标记防止重复购买，缺少其他防刷措施：
- 没有验证码
- 没有 IP 限制
- 没有用户行为分析

#### 问题6：缺少监控告警

**问题描述**：
缺少秒杀成功率、库存消耗速率等监控指标，无法及时发现异常。

## 二、解决方案

### 2.1 P0：数据库库存同步扣减

**方案**：在 `SeckillOrderConsumer` 中同步扣减数据库库存。

**新增内部接口**：

```java
// MarketingClient.java
@PostMapping("/inner/seckill/{activityId}/deduct-stock")
Result<Boolean> deductSeckillStock(@PathVariable("activityId") Long activityId);

// SeckillInnerController.java
@PostMapping("/inner/seckill/{activityId}/deduct-stock")
public Result<Boolean> deductSeckillStock(@PathVariable Long activityId) {
    int rows = activityMapper.deductStock(activityId);
    return Result.success(rows > 0);
}
```

**消费者实现**：

```java
@Override
public void onMessage(SeckillOrderEvent event) {
    // 幂等性检查
    String idempotentKey = SECKILL_ORDER_KEY_PREFIX + eventId;
    Boolean success = redisTemplate.opsForValue().setIfAbsent(idempotentKey, "1", 24, TimeUnit.HOURS);
    if (Boolean.FALSE.equals(success)) {
        return;
    }

    try {
        // 1. 扣减数据库库存
        deductDatabaseStock(activityId);
        
        // 2. 创建秒杀订单
        createSeckillOrder(event);
        
        // 3. 记录监控指标
        recordMetrics(activityId, true);
    } catch (Exception e) {
        // 删除幂等性Key，允许重试
        redisTemplate.delete(idempotentKey);
        
        // 回滚Redis库存和用户购买标记
        rollbackRedis(activityId, userId);
        
        // 记录失败指标
        recordMetrics(activityId, false);
        
        throw e;
    }
}
```

### 2.2 P0：活动结束后完整清理Redis

**方案**：清理库存key、用户购买标记key、时间戳key。

**清理脚本**：

```lua
-- SECKILL_CLEANUP_SCRIPT
local count = 0
if redis.call('EXISTS', KEYS[1]) == 1 then
    redis.call('DEL', KEYS[1])
    count = count + 1
end
if redis.call('EXISTS', KEYS[2]) == 1 then
    redis.call('DEL', KEYS[2])
    count = count + 1
end
return count
```

**Java 实现**：

```java
private void cleanupEndedActivityRedis(Long activityId) {
    // 1. 清理库存和购买标记
    String stockKey = MarketingConstants.SECKILL_STOCK_KEY_PREFIX + activityId;
    String boughtKey = MarketingConstants.SECKILL_USER_BOUGHT_PREFIX + activityId;
    redisTemplate.execute(seckillCleanupScript, List.of(stockKey, boughtKey));
    
    // 2. 清理所有时间戳key
    cleanupTimeKeys(activityId);
}

private void cleanupTimeKeys(Long activityId) {
    String pattern = SECKILL_TIME_KEY_PREFIX + activityId + ":*";
    Set<String> keys = redisTemplate.keys(pattern);
    if (keys != null && !keys.isEmpty()) {
        redisTemplate.delete(keys);
    }
}
```

### 2.3 P1：添加防刷机制

**方案**：IP限流 + 验证码 + 用户行为分析。

**防刷服务接口**：

```java
public interface SeckillAntiFraudService {
    // IP限流
    boolean isIpLimited(String ip, Long activityId);
    void recordIpAccess(String ip, Long activityId);
    
    // 验证码
    boolean verifyCaptcha(String captchaKey, String captchaCode);
    String generateCaptchaKey(Long userId, Long activityId);
    boolean needCaptcha(Long userId, Long activityId);
    
    // 用户行为
    void recordUserFail(Long userId, Long activityId);
    void clearUserFail(Long userId, Long activityId);
}
```

**防刷策略**：

| 策略 | 说明 | 阈值 |
|------|------|------|
| IP限流 | 同一IP在60秒内请求超过10次则限制 | 10次/分钟 |
| 验证码 | 用户连续失败3次后需要验证码 | 3次 |
| 失败记录 | 记录用户失败次数，用于判断是否需要验证码 | 1小时过期 |

**实现代码**：

```java
@Service
@RequiredArgsConstructor
public class SeckillAntiFraudServiceImpl implements SeckillAntiFraudService {

    private static final int IP_LIMIT_THRESHOLD = 10;        // IP限流阈值
    private static final int IP_LIMIT_WINDOW_SECONDS = 60;   // IP限流时间窗口
    private static final int CAPTCHA_FAIL_THRESHOLD = 3;     // 需要验证码的失败次数

    @Override
    public boolean isIpLimited(String ip, Long activityId) {
        String key = IP_LIMIT_PREFIX + activityId + ":" + ip;
        String count = redisTemplate.opsForValue().get(key);
        return count != null && Integer.parseInt(count) > IP_LIMIT_THRESHOLD;
    }

    @Override
    public boolean needCaptcha(Long userId, Long activityId) {
        String failKey = USER_FAIL_PREFIX + activityId + ":" + userId;
        String failCount = redisTemplate.opsForValue().get(failKey);
        return failCount != null && Integer.parseInt(failCount) >= CAPTCHA_FAIL_THRESHOLD;
    }
}
```

### 2.4 P1：添加监控告警

**方案**：使用Redis记录监控指标，定时任务计算并告警。

**监控指标**：

| 指标 | Key模式 | 说明 |
|------|---------|------|
| 秒杀成功数 | `seckill:metrics:{activityId}:{date}:success` | 当天成功次数 |
| 秒杀总请求数 | `seckill:metrics:{activityId}:{date}:total` | 当天总请求次数 |
| 库存消耗速率 | `seckill:metrics:{activityId}:stock:rate` | 库存消耗速率 |

**记录指标**：

```java
private void recordMetrics(Long activityId, boolean success) {
    String today = LocalDate.now().toString();
    
    String successKey = SECKILL_METRICS_PREFIX + activityId + ":" + today + ":success";
    String totalKey = SECKILL_METRICS_PREFIX + activityId + ":" + today + ":total";
    
    redisTemplate.opsForValue().increment(totalKey);
    if (success) {
        redisTemplate.opsForValue().increment(successKey);
    }
    
    redisTemplate.expire(successKey, 7, TimeUnit.DAYS);
    redisTemplate.expire(totalKey, 7, TimeUnit.DAYS);
}
```

**监控告警逻辑**：

```java
@Scheduled(cron = "0 */5 * * * ?")  // 每5分钟执行
public void checkSeckillMetrics() {
    // 1. 计算成功率
    String today = LocalDate.now().toString();
    List<SeckillActivity> activities = activityMapper.selectOngoingActivities();
    
    for (SeckillActivity activity : activities) {
        String successKey = SECKILL_METRICS_PREFIX + activity.getId() + ":" + today + ":success";
        String totalKey = SECKILL_METRICS_PREFIX + activity.getId() + ":" + today + ":total";
        
        String successCount = redisTemplate.opsForValue().get(successKey);
        String totalCount = redisTemplate.opsForValue().get(totalKey);
        
        if (totalCount != null && Long.parseLong(totalCount) > 10) {
            double successRate = Double.parseDouble(successCount != null ? successCount : "0") 
                    / Double.parseDouble(totalCount);
            
            // 成功率低于50%告警
            if (successRate < 0.5) {
                log.warn("秒杀成功率异常: activityId={}, successRate={}", activity.getId(), successRate);
                // 发送告警通知...
            }
        }
    }
}
```

### 6.1 防刷策略

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        秒杀防刷流程                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │

│                 │                │                                       │
```

## 十、总结

v4 版本主要解决了 v3 遗留的数据一致性、内存泄漏、用户体验、防刷机制、监控告警问题：

1. **数据一致性**：消费者中同步扣减数据库库存，保证 Redis 和数据库一致
2. **内存管理**：活动结束后自动清理所有 Redis 数据，防止内存泄漏
3. **用户体验**：结果查询添加超时处理，避免用户无限等待
4. **防刷机制**：IP限流 + 验证码 + 用户行为分析
5. **监控告警**：秒杀成功率、库存消耗速率等监控指标
6. **代码质量**：修复接口注释，添加完善的错误处理

这些优化使秒杀服务更加健壮，适合校园二手交易平台的实际场景。
