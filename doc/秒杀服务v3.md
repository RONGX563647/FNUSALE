本文档记录了秒杀服务从 v2 到 v4 的优化过程，包括问题分析、解决方案、代码实现细节和经验总结。

| 版本 | 主要更新 |
|------|----------|

## 一、问题分析

### 1.1 P0 级别问题（严重）

#### 问题1：Lua 返回值语义混淆（导致少卖）

**问题描述**：
Lua 脚本 `SECKILL_ATOMIC_SCRIPT` 中，当库存为 1 时用户抢购成功，`DECR` 操作后返回 `0`。但 Java 代码将 `result == 0` 误判为"库存不足"。

**问题代码**：

```java
// 错误处理
if (result == 0) {
    throw new BusinessException(5004, "秒杀库存不足");
}
```

**Lua 脚本执行流程**（库存=1 时）：

```lua
local remain = tonumber(stock)  -- remain = 1
if remain <= 0 then return 0 end  -- 1 <= 0? false, 继续

local queueNumber = redis.call('DECR', KEYS[1])  -- 1 - 1 = 0
if queueNumber < 0 then ... end  -- 0 < 0? false, 继续

redis.call('SADD', KEYS[2], ARGV[1])  -- 标记用户已购买
return queueNumber  -- 返回 0
```

**严重后果**：

- 最后一个库存被成功扣减
- 用户被 Redis 标记为"已购买"
- **但 Java 抛出异常，用户被告知失败**
- 导致**少卖（underselling）**问题

---

#### 🔬 问题1详细技术分析

**1. 问题具体表现**

当库存为 **1** 时，用户抢购成功，但系统却告知用户"库存不足"，导致**少卖（underselling）**问题。

**2. 根本原因分析**

**Lua 脚本返回值与 Java 处理逻辑的语义不匹配**：

```
Lua 脚本实际行为：
- 库存=1 时，DECR 返回 0（表示扣减后剩余 0）
- 这代表"成功抢到最后一件"

Java 代码错误理解：
- result == 0 被误判为"库存不足"
- 导致成功抢购的用户被拒绝
```

**3. 技术原理详解**

**Redis DECR 命令的工作原理**：

```
DECR 命令执行过程：
1. 读取当前值：stock = 1
2. 执行减法：1 - 1 = 0
3. 写入新值：stock = 0
4. 返回新值：return 0

关键点：DECR 返回的是操作后的值，不是操作是否成功的布尔值
```

**Lua 脚本执行流程（库存=1 时）**：

```lua
-- 步骤1：获取库存
local stock = redis.call('GET', KEYS[1])  -- stock = "1"

-- 步骤2：检查库存是否充足
local remain = tonumber(stock)  -- remain = 1
if remain <= 0 then 
    return 0  -- 1 <= 0? false，不执行此分支
end

-- 步骤3：扣减库存
local queueNumber = redis.call('DECR', KEYS[1])  -- 1 - 1 = 0
if queueNumber < 0 then
    redis.call('INCR', KEYS[1])
    return 0  -- 0 < 0? false，不执行此分支
end

-- 步骤4：标记用户已购买
redis.call('SADD', KEYS[2], ARGV[1])

-- 步骤5：返回排队号
return queueNumber  -- 返回 0
```

**问题发生的完整时序**：

```
时间线：
T1: 用户发起秒杀请求
T2: Lua 脚本执行成功
    - Redis 库存从 1 变为 0
    - 用户被添加到"已购买集合"
    - 返回值 = 0
T3: Java 代码判断
    - if (result == 0) → true
    - throw new BusinessException("秒杀库存不足")
T4: 用户收到"库存不足"错误

结果：
- Redis 状态：库存=0，用户已标记购买
- 数据库状态：无订单记录
- 用户感知：秒杀失败
- 实际情况：用户本应成功抢到最后一件
```

**4. 解决方案**

**核心思路**：统一返回值语义，使用**正数/零表示成功，负数表示失败**。

| 返回值 | 新语义 |
|--------|--------|
| `>= 0` | 秒杀成功，返回排队号（0 表示抢到最后一件） |
| `-1` | 参数错误 |
| `-2` | 用户已购买 |
| `-3` | 预热值无效 |
| `-4` | 库存不足 |

**5. 代码实现细节**

**Lua 脚本修改**：

```lua
-- 修改前
if remain <= 0 then
    return 0  -- 问题：与成功抢到最后一件的返回值冲突
end

if queueNumber < 0 then
    redis.call('INCR', KEYS[1])
    return 0  -- 问题：同上
end

-- 修改后
if remain <= 0 then
    return -4  -- 明确表示库存不足
end

if queueNumber < 0 then
    redis.call('INCR', KEYS[1])
    return -4  -- 明确表示库存不足
end
```

**Java 处理修改**：

```java
// 修改前
if (result == null || result == 0) {
    throw new BusinessException(5004, "秒杀库存不足");
}

// 修改后
if (result == null || result < 0) {
    String errorMessage = getSeckillErrorMessage(result);
    if (result != null && result == -2) {
        throw new BusinessException(5005, errorMessage);
    }
    throw new BusinessException(5004, errorMessage);
}
// result >= 0：秒杀成功
Long queueNumber = result;
```

**错误消息映射方法**：

```java
private String getSeckillErrorMessage(Long result) {
    if (result == null) return "系统繁忙";
    return switch (result.intValue()) {
        case -1 -> "参数错误";
        case -2 -> "您已参与过该秒杀";
        case -3 -> "预热值无效";
        case -4 -> "秒杀库存不足";
        default -> "系统繁忙";
    };
}
```

#### 问题2：Redis 与数据库不一致

**问题描述**：
Redis 操作成功后，如果数据库操作失败，没有回滚 Redis 的机制。

**问题代码**：

```java
// Redis 操作成功
Long result = redisTemplate.execute(...);

try {
    // 数据库操作可能失败
    localMessageMapper.insert(localMessage);
} catch (Exception e) {
    // 没有 Redis 回滚！
    throw new BusinessException(500, "系统繁忙");
}
```

**严重后果**：

- Redis 库存已扣减
- 用户已被标记购买
- 但数据库没有记录
- 用户无法查询到订单

---

#### 🔬 问题2详细技术分析

**1. 问题具体表现**

Redis 库存扣减成功后，如果数据库操作失败，没有回滚 Redis 的机制，导致：
- Redis 库存已扣减
- 用户已被标记购买
- 但数据库没有记录
- 用户无法查询到订单

**2. 根本原因分析**

**Redis 操作在数据库事务范围外，数据库回滚不会自动回滚 Redis**。

**3. 技术原理详解**

**Spring 事务与 Redis 的隔离性**：

```
Spring @Transactional 事务范围：
┌─────────────────────────────────────────┐
│ BEGIN TRANSACTION                        │
│   └─ 数据库操作（受事务保护）             │
│   └─ 异常时自动回滚                       │
│ COMMIT / ROLLBACK                        │
└─────────────────────────────────────────┘

Redis 操作：
┌─────────────────────────────────────────┐
│ Redis 操作（事务范围外）                  │
│   └─ 不受 Spring 事务管理                │
│   └─ 数据库回滚时不会自动回滚             │
└─────────────────────────────────────────┘
```

**问题场景时序图**：

```
时间线：
T1: Redis Lua 脚本执行成功
    - 库存扣减：50 → 49
    - 用户标记：已购买

T2: 开始数据库事务
    - 尝试插入本地消息表

T3: 数据库操作失败
    - 原因：连接超时/约束违反/磁盘满等
    - Spring 事务回滚

T4: Redis 状态未恢复
    - 库存仍为 49（少 1）
    - 用户仍标记为已购买

T5: 用户查询订单
    - 数据库无记录
    - Redis 显示已购买
    - 用户困惑：明明显示已购买，为什么没订单？
```

**为什么 Redis 不受 Spring 事务管理？**

```
Spring 事务管理原理：
- 基于 AOP 代理
- 只管理实现了 TransactionAware 接口的资源
- 主要是 JDBC Connection、JPA EntityManager

Redis 连接：
- RedisTemplate 使用 RedisConnection
- 不实现 TransactionAware 接口
- 不参与 Spring 声明式事务
```

**4. 解决方案**

**方案**：实现**原子回滚机制**，使用 Lua 脚本保证回滚操作的原子性。

**5. 代码实现细节**

**新增 SECKILL_ROLLBACK_SCRIPT**：

```lua
-- 秒杀回滚脚本（原子操作）
-- KEYS[1]: 库存key
-- KEYS[2]: 用户已购买集合key
-- ARGV[1]: 用户ID
-- 返回值: 1=回滚成功, 0=用户未在购买集合中

-- 1. 检查用户是否在购买集合中
if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 0 then
    return 0  -- 用户未购买，无需回滚
end

-- 2. 移除购买标记
redis.call('SREM', KEYS[2], ARGV[1])

-- 3. 恢复库存
redis.call('INCR', KEYS[1])

return 1
```

**Java 回滚方法实现**：

```java
private void rollbackRedis(Long activityId, Long userId) {
    try {
        String stockKey = MarketingConstants.SECKILL_STOCK_KEY_PREFIX + activityId;
        String boughtKey = MarketingConstants.SECKILL_USER_BOUGHT_PREFIX + activityId;

        Long result = redisTemplate.execute(
                seckillRollbackScript,
                List.of(stockKey, boughtKey),
                userId.toString()
        );

        if (result != null && result == 1) {
            log.warn("Redis 回滚成功: activityId={}, userId={}", activityId, userId);
        } else {
            log.warn("Redis 回滚跳过（用户未在购买集合中）: activityId={}, userId={}", activityId, userId);
        }
    } catch (Exception e) {
        log.error("Redis 回滚失败: activityId={}, userId={}", activityId, userId, e);
    }
}
```

**在 joinSeckill 中集成回滚**：

```java
// 序列化失败时回滚
try {
    messageContent = objectMapper.writeValueAsString(orderEvent);
} catch (Exception e) {
    rollbackRedis(activityId, userId);  // 回滚 Redis
    throw new BusinessException(500, "系统繁忙，请稍后重试");
}

// 数据库插入失败时回滚
try {
    localMessageMapper.insert(localMessage);
} catch (Exception e) {
    rollbackRedis(activityId, userId);  // 回滚 Redis
    throw new BusinessException(500, "系统繁忙，请稍后重试");
}
```

#### 问题4：rollbackRedis 非原子操作

**问题描述**：
回滚 Redis 时使用两个独立的 Redis 命令，非原子操作。

**问题代码**：

```java
// 恢复库存
redisTemplate.opsForValue().increment(stockKey);
// 移除购买标记
redisTemplate.opsForSet().remove(boughtKey, userId.toString());
```

**风险**：
- increment 成功，remove 失败 → 库存恢复，但用户仍显示"已购买"
- remove 成功，increment 失败 → 用户标记清除，但库存未恢复

---

#### 🔬 问题4详细技术分析

**1. 问题具体表现**

回滚 Redis 时使用两个独立的 Redis 命令，非原子操作。

**2. 根本原因分析**

**原实现使用两个独立的 Redis 命令，中间可能被其他操作打断**。

**3. 技术原理详解**

**非原子操作的风险场景**：

```
原实现：
redisTemplate.opsForValue().increment(stockKey);  // 操作1
redisTemplate.opsForSet().remove(boughtKey, userId.toString());  // 操作2

风险场景1：操作1成功，操作2失败
T1: increment 成功 → 库存恢复
T2: remove 失败 → 用户标记未清除
结果：库存正确，但用户仍显示"已购买"

风险场景2：操作2成功，操作1失败
T1: remove 成功 → 用户标记清除
T2: increment 失败 → 库存未恢复
结果：用户可重新购买，但库存少1

风险场景3：并发干扰
线程A: increment → 库存 49 → 50
线程B: decrement → 库存 50 → 49  ← 在两次操作之间插入
线程A: remove → 用户标记清除
结果：库存状态混乱
```

**4. 解决方案**

使用 Lua 脚本保证回滚操作的原子性（已在问题2中详细说明）。

#### 问题8：缓存穿透风险（P1）

**问题描述**：
恶意请求大量不存在的商品ID，绕过Redis直接查询数据库，导致数据库压力过大。

**问题场景**：
```
正常请求：商品ID=1（存在）
→ Redis有缓存 → 直接返回

恶意请求：商品ID=999999（不存在）
→ Redis无缓存 → 查询数据库 → 返回null
→ 大量恶意请求 → 数据库压力骤增
```

**v4解决方案**：布隆过滤器 + 空对象缓存

**商品查询时集成布隆过滤器**：

```java
private ProductVO getProductWithCache(Long productId) {
    // 1. 布隆过滤器检查（快速判断商品是否"可能存在"）
    if (!seckillProductBloomFilter.contains(productId)) {
        log.debug("布隆过滤器过滤: productId={}", productId);
        return null;  // 商品一定不存在
    }

    // 2. 查询Redis缓存
    String cacheKey = PRODUCT_CACHE_KEY + productId;
    ProductVO cached = redisTemplate.opsForValue().get(cacheKey);
    if (cached != null) {
        return cached;
    }

    // 3. 查询数据库
    Product product = productMapper.selectById(productId);
    if (product == null) {
        // 空对象缓存（防止缓存穿透）
        redisTemplate.opsForValue().set(cacheKey, ProductVO.empty(), 5, TimeUnit.MINUTES);
        return null;
    }

    // 4. 写入缓存
    ProductVO vo = ProductVO.from(product);
    redisTemplate.opsForValue().set(cacheKey, vo, 30, TimeUnit.MINUTES);
    return vo;
}
```

## 四、关键代码

### 4.1 SECKILL_ATOMIC_SCRIPT 核心逻辑

```lua
-- 1. 检查用户是否已购买（防重复）
if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then
    return -2
end

-- 2. 获取或初始化库存（懒预热）
local stock = redis.call('GET', KEYS[1])
if stock == false then
    local preloadStock = tonumber(ARGV[2])
    if not preloadStock or preloadStock <= 0 then
        return -3
    end
    redis.call('SET', KEYS[1], preloadStock)
    redis.call('EXPIRE', KEYS[1], ARGV[3])
    stock = preloadStock
end

-- 3. 检查库存
local remain = tonumber(stock)
if remain <= 0 then
    return -4
end

-- 4. 扣减库存
local queueNumber = redis.call('DECR', KEYS[1])
if queueNumber < 0 then
    redis.call('INCR', KEYS[1])  -- 回滚
    return -4
end

-- 5. 标记已购买
redis.call('SADD', KEYS[2], ARGV[1])
redis.call('EXPIRE', KEYS[2], ARGV[3])

return queueNumber  -- >= 0 表示成功
```

**为什么需要第4步的回滚？**

```
并发场景：
T1: 线程A检查库存 remain=1 > 0，通过
T2: 线程B检查库存 remain=1 > 0，通过
T3: 线程A执行 DECR → queueNumber=0，成功
T4: 线程B执行 DECR → queueNumber=-1，失败！

如果不回滚：
- 库存变成 -1（超卖）
- 线程B仍然标记为已购买

回滚后：
- INCR 将库存恢复到 0
- 返回 -4 表示失败
- 线程B不会被标记为已购买
```

## 五、经验总结

### 5.1 Lua 脚本返回值设计原则

**教训**：

- 返回值语义必须清晰，避免歧义
- **不要使用 `0` 同时表示"成功"和"失败"两种状态**
- 建议使用正数表示成功，负数表示失败

**正确做法**：

```
正数/0  → 成功
负数    → 失败（不同负数代表不同错误类型）
```

### 5.2 Redis 与数据库一致性

**教训**：

- Redis 操作在事务范围外，数据库回滚不会自动回滚 Redis
- 必须实现补偿机制
- 回滚操作本身也需要保证原子性

**解决方案**：

1. **原子回滚**：使用 Lua 脚本保证回滚操作的原子性
2. **本地消息表**：确保最终一致性
3. **定时任务补偿**：处理异常情况

### 5.3 秒杀场景架构选择

**校园场景特点**：

- 用户量有限（≤10万）
- 库存通常较少（≤50件）
- QPS 峰值可控（≤500）

**架构选择**：

- Redis + Lua 原子操作是最佳实践
- 无需分布式锁（单机 Redis 足够）
- 异步创建订单提升响应速度

### 5.4 代码审查要点

**秒杀代码必须检查**：

1. Lua 脚本返回值与 Java 处理逻辑是否匹配
2. 是否有 Redis 回滚机制
3. 回滚操作是否原子
4. 边界条件：库存为 1 时是否正确处理
5. 并发场景：是否防止超卖和重复购买

### 5.5 v4 新增：缓存穿透防护

**布隆过滤器使用要点**：

- 布隆过滤器判断"不存在"则一定不存在，判断"存在"则可能存在
- 需要配合空对象缓存使用，防止误判导致穿透
- 定期重建过滤器以更新数据

**注意事项**：

```
布隆过滤器误判场景：
- 过滤器判断商品ID可能存在
- 但实际数据库中不存在
- 需要空对象缓存兜底

解决方案：
- 空对象缓存时间较短（5分钟）
- 避免长时间占用内存
```

### 5.6 v4 新增：监控告警最佳实践

**告警分级原则**：

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| info | 提示信息 | 记录日志，无需立即处理 |
| warning | 警告 | 需要关注，可能影响用户体验 |
| critical | 严重 | 需要立即处理，影响系统可用性 |

**关键告警指标**：

1. **库存告警**：提前预警，避免用户无法购买
2. **性能告警**：响应时间过长影响用户体验
3. **错误率告警**：系统稳定性问题
4. **回滚告警**：可能存在数据一致性问题

### Q1: v3 版本修复的"Lua 返回值语义混淆"问题，在面试中如何回答？

**答案**:

**问题描述**：当库存为 1 时，用户抢购成功，但 Java 代码误判为"库存不足"，导致少卖。

**面试回答模板**：

```
我在项目中遇到过 Lua 脚本返回值语义混淆的问题：

1. 问题现象：
   - 库存为 1 时，用户抢购成功但被告知失败
   - Redis 状态正确（库存已扣减，用户已标记）
   - 但用户收到"库存不足"的错误

2. 根本原因：
   - Redis DECR 命令返回操作后的值
   - 库存=1 时，DECR 返回 0（表示扣减后剩余 0）
   - Java 代码将 result == 0 误判为"库存不足"
   - 实际上 0 表示"成功抢到最后一件"

3. 解决方案：
   - 统一返回值语义：>= 0 表示成功，< 0 表示失败
   - 库存不足返回 -4，而不是 0
   - Java 代码判断 result < 0 才是失败

4. 经验教训：
   - 返回值语义必须清晰，避免歧义
   - 不要用同一个值表示多种含义
   - 边界条件（库存=1）必须测试覆盖
```

### Q2: v3 版本的"Redis 与数据库不一致"问题，如何保证数据一致性？

**答案**:

**问题分析**：

```
场景：
T1: Redis Lua 脚本执行成功，库存扣减
T2: 数据库操作失败（连接超时/约束违反）
T3: Spring 事务回滚
T4: Redis 状态未恢复

结果：
- Redis 库存已扣减
- 用户已标记购买
- 数据库无记录
- 数据不一致！
```

**为什么 Redis 不受 Spring 事务管理**：

```
Spring 事务管理原理：
- 基于 AOP 代理
- 只管理实现了 TransactionAware 接口的资源
- 主要是 JDBC Connection、JPA EntityManager

Redis 连接：
- RedisTemplate 使用 RedisConnection
- 不实现 TransactionAware 接口
- 不参与 Spring 声明式事务
```

**解决方案**：

```java
// 失败时原子回滚
try {
    localMessageMapper.insert(localMessage);
} catch (Exception e) {

    rollbackRedis(activityId, userId);

// Lua 回滚脚本
private static final String ROLLBACK_SCRIPT = """
    if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 0 then
        return 0
    end
    redis.call('SREM', KEYS[2], ARGV[1])
    redis.call('INCR', KEYS[1])
    return 1
    """;
```

### Q3: 为什么回滚操作也要用 Lua 脚本保证原子性？

**答案**:

**非原子回滚的风险**：

```
原实现（两个独立命令）：
redisTemplate.opsForValue().increment(stockKey);    // 命令1
redisTemplate.opsForSet().remove(boughtKey, userId); // 命令2

风险场景1：命令1成功，命令2失败
- 库存恢复，但用户标记未清除
- 用户仍显示"已购买"，无法再次购买

风险场景2：命令2成功，命令1失败
- 用户标记清除，但库存未恢复
- 用户可重新购买，但库存少 1

风险场景3：并发干扰
线程A: increment → 库存 49 → 50
线程B: decrement → 库存 50 → 49  ← 中间插入
线程A: remove → 用户标记清除
- 数据状态混乱
```

**Lua 脚本原子回滚**：

```lua
-- 整个脚本原子执行，不会被其他命令打断
if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 0 then
    return 0
end
redis.call('SREM', KEYS[2], ARGV[1])  -- 移除购买标记
redis.call('INCR', KEYS[1])            -- 恢复库存
return 1
```

### Q4: 如何设计 Lua 脚本的返回值，避免语义混淆？

**答案**:

**设计原则**：

```
┌─────────────────────────────────────────────────────────┐
│              Lua 脚本返回值设计原则                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 语义清晰原则                                        │
│     - 同一个值不要表示多种含义                          │
│     - ❌ 0 同时表示"成功"和"失败"                       │
│     - ✅ >=0 成功，<0 失败                              │
│                                                         │
│  2. 正数成功原则                                        │
│     - 正数/0 表示成功，可携带业务数据                   │
│     - 负数表示失败，不同负数区分错误类型                │
```

```java
void testConcurrentSeckill() throws InterruptedException {
    // 并发测试：100 个线程抢 10 件商品
    int threadCount = 100;
    int stock = 10;
    CountDownLatch latch = new CountDownLatch(threadCount);
    AtomicInteger successCount = new AtomicInteger(0);
    
    for (int i = 0; i < threadCount; i++) {
        new Thread(() -> {
            try {
                Long result = seckillService.joinSeckill(userId, activityId);
                if (result >= 0) {
                    successCount.incrementAndGet();
                }
            } finally {
                latch.countDown();
            }
        }).start();
    }
    
    latch.await();
    
    // 验证成功数等于库存数
    assertEquals(stock, successCount.get());
    // 验证最终库存为 0
    assertEquals("0", redisTemplate.opsForValue().get("seckill:stock:" + activityId));
```

```
- Java 判断：result < 0（正确逻辑）
- 失败时回滚：~0.5ms（额外开销，但失败率低）

总体影响：可忽略
- 正常流程性能无变化
- 失败流程增加一次 Redis 调用
- 但失败率极低（< 0.1%），影响可忽略
```

### Q9: 布隆过滤器如何防止缓存穿透？

**答案**:

**缓存穿透问题**：

```
场景：恶意请求大量不存在的商品ID
正常流程：Redis → 数据库 → 返回
问题：每次都穿透到数据库，造成压力

解决方案：
1. 布隆过滤器：快速判断商品"可能存在"或"一定不存在"
2. 空对象缓存：缓存null值，防止重复穿透
```

**布隆过滤器原理**：

```
特点：
- 空间效率高：10000个元素仅需约18KB
- 查询速度快：O(k)，k为哈希函数数量
- 有误判率：可能判断存在但实际不存在
- 无漏判：判断不存在则一定不存在

工作流程：
1. 添加元素：通过k个哈希函数计算位置，标记为1
2. 查询元素：检查k个位置是否都为1
3. 判断结果：
   - 所有位置都为1 → "可能存在"
   - 有位置为0 → "一定不存在"
```

**本项目实现**：

```java
// 1. 布隆过滤器检查
if (!seckillProductBloomFilter.contains(productId)) {
    return null;  // 一定不存在，直接返回
}

// 2. 可能存在，查询缓存
ProductVO cached = redisTemplate.opsForValue().get(cacheKey);
if (cached != null) {
    return cached;
}

// 3. 查询数据库
Product product = productMapper.selectById(productId);
if (product == null) {
    // 空对象缓存（防止误判导致的穿透）
    redisTemplate.opsForValue().set(cacheKey, ProductVO.empty(), 5, TimeUnit.MINUTES);
    return null;
}
```

### Q11: 活动结束后如何清理Redis数据？

**答案**:

**问题分析**：

```
活动结束后Redis中遗留的数据：
- 库存数据：seckill:stock:{activityId}
- 用户购买标记：seckill:user:bought:{activityId}

不清理的影响：
- 内存占用持续增长
- 可能影响后续活动（相同ID）
```

**解决方案**：

```lua
-- SECKILL_CLEANUP_SCRIPT
local count = 0
if redis.call('EXISTS', KEYS[1]) == 1 then
    redis.call('DEL', KEYS[1])
    count = count + 1
end
if redis.call('EXISTS', KEYS[2]) == 1 then
    redis.call('DEL', KEYS[2])
    count = count + 1
end
return count
```

**调用时机**：

```java
// 方案1：定时任务扫描已结束活动
@Scheduled(fixedRate = 300000)  // 每5分钟执行
```
